# keynote-recap 全量需求 Checklist（项目初心文档）

> 本文是从一次完整的 Google I/O '26 Keynote 简报项目（2026-05-20 ~ 2026-05-21）反向提炼的。
> 38 条用户消息 + 现有产物 + 隐式工程偏好，全部沉淀于此。
> **未来设计争议时，回到这里溯源**。

---

## 0. 项目一句话定义

**输入一个 YouTube/Bilibili 发布会链接，输出一份与本次 I/O '26 Keynote 简报（781 行 / 47 图）同等质量的中文图文复盘报告，可直接粘贴进飞书文档。**

## A. 显式需求（用户原话，逐条溯源）

### A1. 起点诉求（msg #1, #3）

| ID | 原话 | 工程含义 |
|---|---|---|
| A1.1 | "做简报时常需要从**发布会的直播视频**中进行关键帧的截图" | 输入是发布会长视频（1-3 小时） |
| A1.2 | "目前的 AI **很难识别关键帧**，识别后**也很难准确对应到相关内容**" | 必须解决「识别准 + 对应准」两个难点 |
| A1.3 | "自动转写 youtube 和 bilibili 的视频链接，并截取关键帧" | YouTube + Bilibili 双平台 |

### A2. 关键帧定义（msg #7）

| ID | 原话 | 工程含义 |
|---|---|---|
| A2.1 | "关键帧应该是对**样品的展示和关键 PPT 页面的展示**" | 截图目标 = PPT + 产品 demo + 数据图 |
| A2.2 | "**演讲者本身的变化没有那么重要**" | 必须排除演讲者特写 / 远景会场 / 过场 |

### A3. 简报内容要求（msg #9, #12, #18）

| ID | 原话 | 工程含义 |
|---|---|---|
| A3.1 | "现在开始做简报生成，也即 **AI 总结的板块**" | 不止截图，要生成结构化文章 |
| A3.2 | "去**发布官方渠道**查看相关的发布文章或者描述" | 必须联网交叉补充 |
| A3.3 | "但**需要表明是补充的**" | 三种来源（字幕 / 联网 / 判断）严格区分 |
| A3.4 | "**叙事顺序按照发布的重要性和优先级**" | 章节按重要性而非时间序 |

| ID | 原话 | 工程含义 |

| A4.2 | "可以是 html，也可以是其他的" | 双交付：`report.md` + `report.html` |
| A4.3 | "整体概要**整体要用高亮块**" | `<div class="callout">` 包裹 |

### A5. 整体概要方法论（msg #22, #23）

| ID | 原话 | 工程含义 |
|---|---|---|
| A5.1 | "整体概要应该是**基于正文的整理**，进行的**整体概括**" | 概要是正文写完后的提炼，不是大纲 |
| A5.2 | "让读者能够**通过整体概要迅速抓到重点**" | 概要必须可独立阅读、信息密度极高 |
| A5.3 | "**关键判断可以删除**，只需要在**正文每个板块里整体给出洞察**，并在**最前面的关键概要有体现**" | 「核心判断」浓缩进概要 + 文末「一点观察」整章 |
| A5.4 | "整体的**内容层级**有问题——不同 bullet point 层级和数字需要层级" | L1 中文数字 / L2 阿拉伯数字 / L3 段落标签 + 4 空格缩进 |

### A6. 筛图三原则（msg #29，**最重要的工程沉淀**）

| ID | 原话 | 工程含义 |
|---|---|---|
| A6.1 | "**有信息量**——目前很多图只是单独展示产品名称或进行转场" | 拒绝 slogan / 转场 / 营销图 |
| A6.2 | "**和所在板块内容直接相关**，起到视觉补充作用" | 图必须能解释或佐证当前板块论点 |
| A6.3 | "**核心内容重复的图删除多余的**" | 同信息严格去重 |

### A7. 图片来源优先级（msg #30）

| ID | 原话 | 工程含义 |

| A7.1 | "**抽帧图质量似乎更高，官方图很多无实际意义**" | 抽帧 > 官方图 |
| A7.2 | "**把抽帧图的优先级提高**" | 官方图只用于抽帧无法替代的场景 |

## B. 底层要求（行为模式中观察到的工程价值观）

### B1. 工程价值观

| ID | 偏好 | 来源证据 | 工程含义 |
|---|---|---|---|
| B1.1 | **宁缺毋滥** | 65→55→46 张反复删图 + frame_scorer.py `--min-score` 阈值设计 | 默认筛图严格，宁少勿多；用户事后补图 cheap |
| B1.2 | **可信溯源** | msg #12 "需要表明是补充的" + 现有 `step3_summarize.py` 三种来源严格区分 | `> 📎 **补充信源**：[URL]` 块格式固定 |
| B1.3 | **不编造** | 补充查证报告专门有「未查到/官方未公布」一节 | Stage 4 必须输出 unknowns 清单；Stage 5 必须保留透明声明节 |
| B1.4 | **真实读图** | 三次发现 caption 错配（msg #27, #32, #35），subagent 跑出的 caption 错配率高 | Stage 5.5 必须 LLM 真的看图核对 caption（不能盲信 Stage 3 输出） |
| B1.5 | **质量优先于速度** | msg #16, #21, #28, #31, #36 都是 "Continue" 信号，无催促 | 不优化跑速，优化质量；可中断 / 检查点不嫌烦 |

## D. 验收标准（quality bar）

用同一个 I/O '26 Keynote 链接 (`https://www.youtube.com/watch?v=wYSncx9zLIU`) 跑产品，输出必须满足：

1. **结构**
   - [ ] 有整体概要 callout（`<div class="callout">`）
   - [ ] 整体概要按发布优先级分 8-12 块
   - [ ] 每块包含主题 + 定调 + 嵌套证据
   - [ ] 整体概要内含「核心判断」子项
   - [ ] 文末有独立「一点观察」整章
   - [ ] 文末有「信源说明」节
   - [ ] 文末有「未查到 / 官方未公布」透明声明

2. **图片**
   - [ ] 总图数 30-60 张
   - [ ] 抽帧图占比 ≥ 60%
   - [ ] **每个 ## 章节至少 1 张图**（A8 硬约束）
   - [ ] 重要章节 4-6 张图，次要 1-2 张
   - [ ] 无 slogan / 转场 / 营销图
   - [ ] 同信息无重复
   - [ ] 每张图 caption 含完整上下文（不依赖原视频也能懂）

3. **联网交叉**
   - [ ] 至少 8 个 `> 📎 **补充信源**：[URL]` 块
   - [ ] 每个补充信源都有可访问 URL

4. **层级**
   - [ ] L1 用 `## 一、…`（中文数字）
   - [ ] L2 用 `### 1.1 …`
   - [ ] L3 用 `**xx**：`
   - [ ] bullet 子项 4 空格缩进

5. **文风**
   - [ ] 无 emoji（除 📎 / ✅）
   - [ ] 无「巨大 / 显著 / 革命性 / 让我们 / 不仅仅是」
   - [ ] 数据用「时间→数据」对比序列
   - [ ] 章节按重要性而非时间序

6. **可读性**
   - [ ] HTML < 15 MB（base64 内嵌后）
   - [ ] 可直接粘贴进飞书文档

**与本次手工产物质量差距 < 15%** 即为达标。

## E. 不在 MVP 范围（明确写出避免 scope creep）

- 多语言输出（英文 / 日文）
- 飞书 / Notion / Confluence 直接发布集成
- 风格 few-shot（用户传入参考文章）
- Web 应用
- PDF 输出
- 字幕实时（流式）处理

## F. 决策不可回滚的部分（固化即承诺）

1. **关键帧定义** = PPT + 产品 demo + 数据图（A2）
2. **筛图三原则** = 信息量 + 相关性 + 去重（A6）
3. **图源优先级** = 抽帧 > 官方图（A7）
4. **每板块至少 1 图** = 硬约束（A8）
5. **三种来源严格区分** = 字幕 / 联网 / 判断（A3.3 + B1.2）
6. **章节按发布优先级而非时间序**（A3.4 + B3.7）
7. **整体概要是正文写完后的提炼**（A5.1）
8. **HTML 必须自包含**（B4.4）
9. **OpenAI 兼容 endpoint，用户自配模型**（B2.3）
10. **中文优先**（B2.4）
11. **包管理 = uv**（M1 决策 2026-05-21）—— 10×+ 速度，2026 年事实标准
12. **CLI = Click**（M1 决策 2026-05-21）—— 与 video-recap 复用一致
13. **测试 = pytest**（M1 决策 2026-05-21）—— 事实标准，pyproject.toml 已配
14. **MVP 只做 keynote-recap 模板**（M1 决策 2026-05-21）—— 不含 talk/interview/tutorial
15. **默认 search provider = DuckDuckGo + webfetch**（M1 决策 2026-05-21）—— 零 key 零成本，必要时用户可换 Tavily
