# keynote-recap 方法论（完整中文说明）

> 本文是 [requirements.md](./requirements.md) 中所有约束的**操作化展开**，是 LLM 在 stage 5（draft）写作时遵循的"显式 + 隐式规则"全集。
>
> **重要**：所有 `prompts/05-draft-*.md` 都会引用本文片段。如需修改风格，先改本文，再 sync 到 prompts。

---

## 0. 总纲

一份合格的 keynote 复盘简报必须满足以下四个层次：

| 层次 | 关注点 |
|---|---|
| L1 结构 | 整体概要 + 主线章节 + 一点观察 + 信源说明 + 透明声明 |
| L2 内容 | 每个论点都有「字幕原话 / 联网补充 / 独立判断」三种来源严格区分 |
| L3 视觉 | 每个章节都有图（A8 硬约束），图都有信息量、有相关性、不重复 |
| L4 文风 | 反 emoji、反夸张、反 AI 套话、数据驱动、表格化并列信息 |

### 1.2 整体概要的写法（A5）

> 来源：requirements.md A5

**关键原则**：

1. **概要是正文写完后的提炼**——不是大纲，不是预先写好的导言。先写正文，再回过头基于正文产出概要。
2. **每块结构固定**：`**<主题>：<一句定调>**` + 嵌套 bullet（证据） + `**核心判断**：<一句话>`
3. **8-12 块**——少了概括不够，多了破坏「迅速抓重点」的目的
4. **按发布优先级排序**——最重要的在最前面
5. **核心判断必须独立**——不是定调的同义重复，是基于该板块的独立分析
6. **整块用 `<div class="callout">` 包裹**

**反例**：
```markdown
- **模型层**
    - 发布了 Gemini 3.5 Flash
    - 速度快
```
**正例**：
```markdown
- **模型层：Gemini 3.5 家族首发 + Google 首个全模态模型 Omni**
    - **Gemini 3.5 Flash**（今日可用）：能力整体超越上一代旗舰 3.1 Pro；速度比其他前沿模型 **4×**、Antigravity 内 **12×**；头部 Cloud 客户切 80% workload 可**年省 $1B**
    - **Gemini Omni Flash**（今日可用）：任意输入→任意输出的全模态模型；首发于 Gemini App / Google Flow / YouTube Shorts
    - **Gemini 3.5 Pro**：下月发布，定位高精度、长链路推理、复杂代码重构
    - **核心判断**：3.5 Flash 是「牺牲知识深度，换取执行能力」——Google 在打吞吐量战争，不是 benchmark 战争
```

### 1.3 章节顺序：按发布优先级（A3.4 + B3.7）

> 来源：msg #18

**禁止**按视频时间序排章节。**必须**按发布的重要性排序：

1. 最重大主线发布（如本次 I/O 的 Agent 体系）
2. 主力开发者平台
3. 模型层
4. ...
5. 收尾（基础设施、订阅价格、安全/科学、交付节奏）

**判断"重要性"的依据**：
- 演讲者花了多少时间讲（粗略指标）
- 是否带来生态级改变（协议层 > 产品层 > 功能层）
- 是否是行业首发或竞争对决信号

### 1.4 「一点观察」整章

> 来源：msg #23

文末必须有独立的「一点观察」章节：

- **6-10 个观察**，每个观察 2-4 段
- **完全独立判断**，不是再总结一遍
- 标题用「这是 X 的转折点」「Y 是这次最被低估的发布」等具体论点形式
- 末尾明确声明「非发布会原话」

## 2. 三种来源严格区分（L2）

> 来源：requirements.md A3.3 + B1.2

### 2.1 三种来源

| 来源类型 | 标记方式 | 何时使用 |
|---|---|---|
| **字幕原话** | 直接写正文 + 必要时 `> "原话"` 引用块 | 演讲者明确说过的内容 |
| **联网补充** | `> 📎 **补充信源**：根据 [Source Name](URL)，<内容>` | 字幕没说全或说得不准的细节（精确版本号、定价、上市日期、benchmark 数字） |
| **独立判断** | 正文段落 + 明确「核心判断 / 我的观察 / 一点观察」标记 | 作者个人分析 |

### 2.2 绝对禁止

**禁止**把联网搜来的细节直接写进正文当作字幕里说过的内容。读者必须能区分：

- 「Sundar 说的话」
- 「字幕没说，但官方博客有的细节」
- 「我作为分析师的判断」

### 2.3 透明声明节（B1.3）

文末必须有独立小节列出：

1. 字幕里**有**但联网**没查到**的（如 Plus / Pro 月费具体数字）
2. 联网渠道**没公布**的（如 Audio Glasses 价格）
3. 字幕里听到了但**判断不准**的（如 Stitch 累计 1 亿 UI 画面这一具体数字）

**这是诚信底线**——AI 必须主动声明自己不知道什么。

## 3. 筛图三原则（L3 - 最重要）

> 来源：requirements.md A6 + msg #29

### 3.1 三原则

| 原则 | 含义 | 反例 |
|---|---|---|
| ① 信息量 | 含可读数据、产品 UI、技术架构图、对比表格、产品照片、合作伙伴 logo 墙 | 纯标题文字「Gemini Omni」/ slogan「Bring any idea to life」/ 品牌 logo 大字「Antigravity 2.0」/ 转场页 |
| ② 相关性 | 与所在板块内容直接相关，起视觉补充作用 | Audio Glasses 章节里放 TPU 芯片图 |
| ③ 去重 | 同信息严格去重 | Fitbit Air 四张模特穿戴图（信息相同） |

### 3.2 图源优先级（A7）

> 来源：msg #30 「抽帧图质量似乎更高」

**抽帧 > 官方图**

- 抽帧图：从视频中按时间戳截取
- 官方图：从厂商博客 / 新闻稿下载

**官方图只用于**：抽帧拍不到的产品形态（产品照片、UI 真机截图、合作伙伴墙）。

### 3.3 每板块至少 1 张图（A8 硬约束）

> 来源：msg #35 「最早提过」

**这是项目最早的硬约束**：

- 每个 `## 一、…` 章节至少 1 张图
- 每个 `### 1.1 …` 子章节最好也有图（不是硬约束，但推荐）
- **重要章节 4-6 张图**，**次要章节 1-2 张**

### 3.4 多图允许（A9）

> 来源：msg #27

如果章节内有多个关键节点（多个产品 demo、多个数据点），允许放多张图。例如：
- Gemini Spark 章节可放 4 张：定位 / email demo / Block Party / 手机 brain-dump
- TPU 章节可放 3 张：演进时间线 / 双芯片实物 / 规格对比表

### 3.5 caption 撰写规范（B1.4）

> 来源：msg #27, #32, #35 反复出现的「caption 错配」教训

**caption 必须含完整上下文，让读者无需查原视频也能理解**：

**反例**：
```markdown
![Omni demo](frames/frame_15.jpg)
```

**正例**：
```markdown
![Omni 「知识驱动生成」实例：演讲者输入 prompt「make a claymation explainer of protein folding」→ Omni 生成黏土风格氨基酸链科普视频帧（标题"AMINO ACIDS"，五彩黏土小球串联成肽链结构）](frames/frame_15.jpg)
```

caption 必须包含：
1. 这是哪个章节的什么环节
2. 演讲者当时在演示什么
3. 图的具体内容（标题文字 / 图表数据 / UI 元素）

## 4. 文风规则（L4）

> 来源：requirements.md B3

### 4.1 反 emoji（B3.2）

**全文几乎无 emoji**，仅以下两个例外允许：
- `📎` 用于「补充信源」标记
- `✅` 用于表格里的「已发布 / 已上线」状态

**禁止**：🔥🚀💡⭐️🎉👀🤔（一律删除）

### 4.2 反夸张词（B3.3）

**禁用词清单**：

| 类别 | 禁用词 |
|---|---|
| 形容词类 | 巨大、显著、革命性、惊人、震撼、重磅、史诗级、颠覆性、空前 |
| 程度副词类 | 大幅、极大、非常、十分（限定语境用「明显」替代） |
| 营销词类 | 让人眼前一亮、令人印象深刻、不容小觑 |

**替代策略**：用具体数字 + 对比代替形容词。
- ❌ 「速度有了巨大提升」
- ✅ 「速度比其他前沿模型 4×，Antigravity 内 12×」

### 4.3 反 AI 套话（B3.4）

**禁用句式**：

- 「让我们一起来看看……」
- 「值得我们关注的是……」
- 「不仅仅是……更是……」
- 「在当今的……时代」
- 「这无疑标志着……」
- 「未来已来……」
- 「以下是关于 X 的……」

**替代**：直接说事实。
- ❌ 「让我们看看 Gemini Spark 的核心能力」
- ✅ 「Gemini Spark 三类典型场景」

### 4.4 数据驱动叙事（B3.1）

**关键数据用「时间→数据」三段对比序列**呈现：

| 维度 | 旧值 | 中间值 | 新值 |
|---|---|---|---|
| Capex | $31B（2022） | — | $180-190B（2026） |
| Token | 9.7T（2024.5） | ~480T（2025.5） | 3,200T（2026.5） |
| Ultra | $250 | — | $200 |

**绝对值 + 倍数 + 同比/环比**至少给两个，让读者能感知变化幅度。

### 4.5 表格优先并列信息（B3.6）

以下情况**必须**用表格：

- 发布清单 / 时间线
- 产品对比 / 价格档位
- 多产品规格对比
- 合作伙伴列表
- 上线时间表
- 三种 / 四种典型场景

### 4.6 引用块用法

`> "..."` 块**只用于演讲者原话**。其他 `>` 块（如「📎 补充信源」「⚠️ 警告」）必须有标识符开头。

## 5. 层级规范（A5.4）

> 来源：msg #23

| 层级 | 写法 | 举例 |
|---|---|---|
| L1 章节 | `## 一、…`（中文数字） | `## 一、Agent 体系（最大主线）` |
| L2 子章节 | `### 1.1 …`（阿拉伯数字） | `### 1.1 Gemini Spark — 24/7 个人 Agent` |
| L3 段落标签 | `**xx**：` | `**定位**：` / `**架构**：` / `**核心判断**：` |
| L4 bullet | `- ` 顶格 | `- 跨数据源汇编邮件` |
| L4.1 bullet 子项 | 4 空格缩进 + `- ` | `    - **/ghostwriter** 个人 skill` |

**图片永远顶格**——不缩进，不在 bullet 内。

## 6. 章节内部模板

### 6.1 标准章节结构

```markdown
## 一、<最重要主题>（如有副标题，加括号说明）

> **定位**：<一两句解释为什么这一章重要>

### 1.1 <子主题 1>

![<完整 caption>](frames/frame_XX.jpg)

**定位**：<这个子项是什么、为什么发布>

<2-3 段正文。每段 2-5 句。短句多于长句。>

> "<演讲者原话>"

**关键数据 / 三大能力 / 上线节奏**：

| 维度 | 数字 / 内容 |
|---|---|
| ... | ... |

> 📎 **补充信源**：根据 [Source](URL)，<联网补充>。

**风险设计 / 隐含意义**（可选）：<一两句>
```

### 6.2 数据章节模板

```markdown
### N.M <数据主题>

![<图说>](frames/frame_XX.jpg)

> "<演讲者原话引用>"

| 时间 | 数字 |
|---|---|
| 2022 | $31B |
| 2026 | $180-190B（约 6×） |

**核心判断**：<一句独立分析>
```

### 6.3 产品章节模板

```markdown
### N.M <产品名> — <定位副标题>

![<产品形态图>](frames/frame_XX.jpg)

**定位**：<面向谁、解决什么问题>

**架构**：

| 层 | 组件 |
|---|---|
| 模型 | ... |
| 执行环境 | ... |
| 数据源 | ... |

**X 类典型场景**：
- **<场景 1>**：<细节>
- **<场景 2>**：<细节>

**上线节奏 + 价格**（联网补充）：

| 时间 | 范围 |
|---|---|
| 本周 | trusted testers |
| 下周 | Beta，仅美国 |

> 📎 **补充信源**：[URL]
```

## 7. 验收 checklist

每写完一份 report.md，按以下顺序验收：

### 7.1 结构（L1）
- [ ] 顶部有 `<div class="callout">` 整体概要
- [ ] 概要 8-12 块，按发布优先级
- [ ] 每块含主题 + 定调 + 嵌套证据 + 核心判断
- [ ] 文末有独立「一点观察」整章（6-10 个观察）
- [ ] 文末有「信源说明」节
- [ ] 文末有「未查到 / 官方未公布」透明声明

### 7.2 来源（L2）
- [ ] 至少 8 个 `> 📎 **补充信源**：[URL]` 块
- [ ] 每个补充信源都有可访问 URL
- [ ] 引用块仅用于演讲者原话或专门标识

### 7.3 视觉（L3）
- [ ] 总图数 30-60 张
- [ ] 抽帧图占比 ≥ 60%
- [ ] **每个 ## 章节至少 1 张图**（A8）
- [ ] 无 slogan / 转场 / 营销图
- [ ] 无重复信息图
- [ ] 每张图 caption 含完整上下文

### 7.4 文风（L4）
- [ ] 无 emoji（除 📎 / ✅）
- [ ] 无禁用形容词 / 副词 / 营销词
- [ ] 无 AI 套话句式
- [ ] 数据用「时间→数据」对比序列
- [ ] 并列信息用表格

### 7.5 层级
- [ ] L1 用 `## 一、…`
- [ ] L2 用 `### 1.1 …`
- [ ] L3 用 `**xx**：`
- [ ] bullet 子项 4 空格缩进
- [ ] 图片顶格

### 7.6 可读性
- [ ] HTML < 15 MB
- [ ] 可直接粘贴进飞书文档
- [ ] 不依赖外部图床（base64 内嵌）

## 8. 持续更新机制

> 来源：B2.2「AI 沉淀方法论」

**用户每次反馈后**：

1. 判断这次反馈是否属于**新约束**或**已有约束的实例**
2. 若是新约束：增补 [requirements.md](./requirements.md) 对应章节，标注会话日期
3. 若是已有约束被违反：检查 `prompts/*.md` 哪一份漏写了规则，补强
4. 让 LLM 在下次类似任务自动应用新规则

**这是项目自我进化的机制**——一次性问题不修，模式问题进 prompts。
