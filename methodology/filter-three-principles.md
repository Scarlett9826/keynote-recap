# 筛图三原则（详解）

> 这是 keynote-recap 最重要的工程沉淀，源自 Google I/O '26 简报项目反复迭代（msg #29 + #30 + #32 + #35）。
> 三轮筛图（65 → 55 → 46 张），每一轮都对应一条原则的强化。

---

## 原则一：信息量

### 必须保留

✅ **可读数据图**：图表中数字清晰可见
- 例：Capex 柱状图（$31B → $180B）
- 例：性能对比表（3.1 Pro vs 3.5 Flash）

✅ **产品 UI / 真机截图**：能看到具体界面元素
- 例：Gemini App 重做后的对话界面
- 例：Antigravity 2.0 的 IDE 截图

✅ **技术架构图**：含组件名、数据流、连接线
- 例：TPU 8t / 8i 双芯片对比规格表
- 例：UCP 协议层次结构

✅ **产品形态实物图**
- 例：Audio Glasses 与 Warby Parker / Gentle Monster 联名产品图
- 例：Fitbit Air 4 色腕带产品矩阵

✅ **合作伙伴 logo 墙**：当 logo 数量本身是信息时
- 例：UCP 24 家伙伴 logo 墙（Shopify / Etsy / Wayfair / ...）

✅ **现场 demo 关键瞬间**
- 例：Black Hole 交互可视化组件运行截图
- 例：Stitch 实时 UI 修改 demo

### 必须拒绝

❌ **纯标题页**："Gemini Omni" 单独一行大字
❌ **营销 slogan**："Bring any idea to life"
❌ **品牌 logo 大字**：占满屏幕的产品 logo
❌ **过场页**：黑屏 + 闪烁 logo / 章节分隔页
❌ **无文字的纯背景**：装饰图 / 抽象艺术

### 灰色地带（需结合上下文判断）

🟡 **产品形态截图但有营销 slogan**：如果 slogan 占比 < 30% 且产品形态清晰，保留
🟡 **远景拍摄但能看到关键内容**：如果舞台远景但 PPT 投影仪屏幕可读，保留

## 原则二：相关性

### 关键判断

**图必须能解释或佐证当前章节的论点**。

### 反例（来自 Google I/O '26 简报教训）

| 章节 | 错误图 | 为什么错 |
|---|---|---|
| Search 章节 | "Gemini Omni" logo 大字 | 与 Search 无关，应放 Omni 章节 |
| Audio Glasses 章节 | TPU 双芯片实物图 | 与硬件章节交叉错位 |
| Antigravity 章节 | Gemini App 对话界面 | 应放 Gemini App 章节 |

### 正确判断流程

```
对每张图问三个问题：
1. 这张图讲的是什么？
2. 当前章节核心论点是什么？
3. 图能否直接支撑论点？

如果答 1 ≠ 答 2 的范围 → 拒绝
如果答 3 是「需要解释才能联系上」→ 拒绝（应放更直接相关的章节）
```

## 原则三：去重

### 严格去重场景

✅ **同信息多视角**：保留信息密度最高的一张
- 例：Fitbit Air 4 张模特图 → 保留 1 张全产品色彩矩阵图

✅ **同产品多角度**：保留正面 + 必要场景图
- 例：Audio Glasses 已有 Warby Parker 产品图 → 删除同款不同角度的产品图

✅ **同 demo 不同帧**：保留信息量最大的一帧
- 例：连续 3 帧 Spark 邮件 demo → 保留含完整 UI 的那帧

### 允许保留

✅ **同主题不同侧面**：如 TPU 章节可同时有「演进时间线」+「双芯片实物」+「规格对比表」
✅ **同产品不同场景**：如 Spark 章节可同时有「桌面 demo」+「手机端」+「Block Party 协作」

## 图源优先级（A7）

> 来源：msg #30

```
抽帧图 > 官方图
```

### 为什么抽帧 > 官方图

1. **抽帧图带演讲者上下文**：可以看到 PPT 状态、演示中的真实交互、临时的对比标记
2. **官方图很多是营销渲染**：常含 slogan、品牌色块、过度渲染的产品图
3. **抽帧图与字幕严格对齐**：第 N 秒字幕说"看这个数字" → 第 N 秒的帧就是这个数字
4. **官方图常滞后**：keynote 当天发布的产品，官方博客图可能延迟一两天

### 官方图唯一应保留的场景

- 抽帧拍不到的产品形态（高质量产品照片）
- 抽帧画面太小看不清的 UI 真机截图
- 合作伙伴 logo 墙（抽帧时角度可能不够正）
- 复杂技术架构图（官方版本更清晰）

## 每板块至少 1 张图（A8 硬约束）

> 来源：msg #35「最早提过」

### 这是项目最早的硬约束

```
∀ 章节 chapter ∈ report.md:
    chapter.images ≥ 1
```

### 失败处理

如果 stage 5 写完后某章节没图：

1. **优先**回 frames/ 找候选——可能 vision filter 误删
2. **其次**联网找官方图——主动查 vendor 博客
3. **最后**才考虑放空白——但要在 caption 里说明（不推荐）

**绝不接受**「这个章节没合适的图」作为理由。

---

## 多图允许（A9）

> 来源：msg #27

### 章节图量自适应

| 章节重要性 | 推荐图数 |
|---|---|
| 主线（如本次的 Agent 体系） | 4-6 张 |
| 重要章节 | 2-4 张 |
| 次要章节 | 1-2 张 |
| 轻量章节 | 1 张（A8 下限） |

### 多图判断信号

✅ 章节有多个独立产品 / demo
✅ 章节有多个时间点的发布（如 Flow 5 项更新）
✅ 章节有多个对比维度（如 TPU 8t / 8i 对比）
✅ 章节有从 A 到 B 的过程展示

❌ 同信息不同角度（去重原则约束）
❌ 装饰性图（信息量原则约束）

## caption 撰写规范

> 来源：msg #27, #32, #35 反复出现的 caption 错配教训 + B1.4

### 反例

```markdown
![Spark](frames/frame_31.jpg)
![demo](frames/frame_32.jpg)
![cost](frames/frame_29.jpg)
```

读者无法从 caption 知道图里讲什么。

### 正例

```markdown
![Gemini Spark — 24/7 个人 AI Operator 概念图，跑在 Google Cloud 专属虚拟机上的可视化](frames/frame_31.jpg)

![Spark email demo：演讲者口述 prompt → Spark 用 /ghostwriter skill 生成符合本人语气的邮件草稿（屏幕显示三栏式 UI：左侧 prompt、中间 draft、右侧来源标注）](frames/frame_32.jpg)

![3.5 Flash 成本叙事图：年省 $1B 的口径——Top Cloud 客户每天 ~1T tokens，混合用 Flash 替代 80% 前沿模型](frames/frame_29.jpg)
```

### caption 必含三要素

1. **是什么**：图的主体（产品 UI / 数据图 / 实物 / 架构）
2. **讲什么**：图传递的核心信息
3. **上下文**：演讲者当时在说什么、为什么放这张图

### 长度参考

- 简单图：30-50 字
- 数据图：50-100 字（含关键数字）
- 复杂 demo：100-150 字（含 UI 元素和交互流程）
