# 三种来源严格区分（信源标注规范）

> 这是 keynote-recap 的诚信底线。读者必须能区分作者听到了什么、查到了什么、判断了什么。
> 源自 msg #12「需要表明是补充的」+ B1.2 + B1.3。

---

## 1. 三种来源

| 类型 | 来源 | 标记方式 | 性质 |
|---|---|---|---|
| ① 字幕原话 | 视频字幕 | 直接写正文，必要时 `> "..."` 引用块 | 客观事实（演讲者说了） |
| ② 联网补充 | 官方博客 / 新闻稿 / 产品页 | `> 📎 **补充信源**：[Source](URL)` 块 | 客观事实（来自第三方） |
| ③ 独立判断 | 作者分析 | 正文段落 + 明确「核心判断 / 我的观察 / 一点观察」标记 | 主观分析 |

---

## 2. 字幕原话

### 写法

```markdown
Demis 上场亲自发布。**from any input create anything**——从任意输入生成任意输出。

> "AGI is now on the horizon and it will be the most profound and impactful technology ever invented." —— Demis Hassabis
```

### 规则

- 演讲者具体说过的话，可以直接写正文
- 关键金句用 `>` 引用块，标注说话人
- 多语种保留原文（英文 keynote 可保留原英文）+ 中文意译

### 反例

❌ 把字幕没说过的细节写进正文：
```markdown
Sundar 宣布 Gemini 3.5 Flash 价格 $0.075/M tokens。  ← 字幕里没说价格！
```

正确做法：
```markdown
Sundar 宣布 Gemini 3.5 Flash 今日可用。

> 📎 **补充信源**：根据 [blog.google](URL)，Flash 定价为 input $0.075/M tokens。
```

## 3. 联网补充

### 标记格式（固定）

```markdown
> 📎 **补充信源**：根据 [<Source Name>](<URL>)，<内容>
```

或

```markdown
> 📎 **补充信源**：[<Source Name>](<URL>)。<内容>
```

### 触发场景

应该联网查证的内容：

✅ 产品 / 功能名称的官方拼写
✅ 版本号、上市日期
✅ 定价（特别是字幕里没说全的）
✅ Benchmark 数字、对比基准
✅ 合作伙伴的官方表述（公司名拼写、合作范围）
✅ 关键术语的官方定义
✅ 上线时间表（dev / beta / GA）
✅ 区域可用性

### 信源优先级

```
1. 厂商官方博客（blog.google / openai.com/blog / anthropic.com/news）
2. 厂商官方产品页 / newsroom
3. 大型科技媒体（TechCrunch / The Verge / Bloomberg）
4. Wikipedia / 维基（仅用于背景信息，不用于发布细节）
5. 社交媒体（X / Reddit）— 不建议（信息可信度低）
```

### 短链允许

URL 太长可以用 markdown 链接缩写：
```markdown
> 📎 **补充信源**：[blog.google/.../sundar-pichai-io-2026](https://blog.google/innovation-and-ai/sundar-pichai-io-2026/)
```

## 4. 独立判断

### 三种允许的标记

#### 4.1 整体概要里的「核心判断」（每块末尾）

```markdown
- **模型层：Gemini 3.5 家族首发 + Google 首个全模态模型 Omni**
    - <证据 bullet>
    - **核心判断**：3.5 Flash 是「牺牲知识深度，换取执行能力」——Google 在打吞吐量战争
```

#### 4.2 章节末尾的「核心判断」（必要时）

```markdown
**核心判断**：UCP 不是产品而是协议——这是这次 I/O 最被低估的发布
```

#### 4.3 文末「一点观察」整章

```markdown
## 十四、一点观察（独立判断，非发布会原话）

### 1. 这是 Google 把 AI 从"功能"变成"基础设施"的转折点

发布会从头到尾都不是在讲单个模型有多强，而是在讲：算力、协议、平台、订阅。
```

### 规则

- **每个独立判断段落必须有明确的语义标识**：「核心判断」/「一点观察」/「我的观察」/「分析」
- **「一点观察」整章末尾必须声明非原话**：
  ```markdown
  ## 十四、一点观察（独立判断，非发布会原话）
  ```

## 5. 透明声明节（B1.3）

### 文末必须有的小节

```markdown
## 信源说明

**字幕原话**：所有引号内文本均为视频字幕逐字摘录。

**📎 补充信源**：以 `> 📎 **补充信源**` 标记的内容，来自字幕外的官方一手渠道交叉验证：

- [URL 1] - 描述
- [URL 2] - 描述
...

**官方未公布 / 本次未查到**（透明声明）：

1. <数据 1>（说明为什么没查到）
2. <数据 2>
...

**我的判断**：整体概要中的「**核心判断**」分句、以及文末「一点观察」整章，全部为本简报作者独立判断，不属于发布会原话。
```

### 「未查到」清单的写法

每条必须说明：

1. **是什么内容**（具体到哪个数据点）
2. **为什么没查到**（官方未公布 / 区域差异 / 信息不一致 / 等）

### 例子

```markdown
**官方未公布 / 本次未查到**（透明声明）：

1. **Google AI Plus / AI Pro 当前月费具体数字** — 官方页面用动态组件按地区渲染，未在 I/O 博客直写
2. **Audio Glasses 的具体售价** — 官方仅给出"sneak peek"，"later this fall" 上市，价格未公布
3. **Display Glasses 的发布时间表** — 官方仅说 Audio 先行，Display 时间表未明示
```

## 6. 绝对禁止

### 禁止行为清单

❌ 把联网搜来的细节直接写进正文当作字幕里说的内容
❌ 给一个不存在或不可访问的 URL
❌ 在 `> 📎` 块里写主观判断（应该用「核心判断」/「一点观察」）
❌ 整片简报没有任何 `> 📎` 块（说明没做联网交叉，违反 A3.2）
❌ 没有「未查到」透明声明节（违反 B1.3）
❌ 「一点观察」整章没有「非发布会原话」声明

---

## 7. 数量约束

### 联网补充最低标准

- **至少 8 个 `> 📎` 块**（一份合格的 keynote 简报）
- 每个章节至少 1 个（重要章节）
- 透明声明节至少 5 条「未查到」

### 信源 URL 数

- 文末「信源说明」节列出的 URL ≥ 6 个
- 每个 URL 都必须可访问

## 8. URL 验证（实施层）

`stages/research.py` 实现时：

1. 每次 LLM 给出 URL，先用 HEAD 请求验证可访问
2. 失败则要求 LLM 重新搜索或标为「未查到」
3. 不允许虚假 URL（hallucinated URL 是常见 LLM 错误）

```python
# 伪代码
def validate_url(url: str) -> bool:
    try:
        r = requests.head(url, timeout=5, allow_redirects=True)
        return r.status_code < 400
    except:
        return False
```
